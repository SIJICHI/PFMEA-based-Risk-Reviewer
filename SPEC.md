# 製剤工程 PFMEA ナレッジ検索エージェント 実装仕様書

## 1. 目的とゴール

製薬工場の固形製剤（錠剤）製造ラインを新たに立ち上げる場面を想定し、工場の技術者が自然言語で
「この工程で注意するべき不良現象は？」のように問い合わせると、過去に蓄積されたPFMEA（工程故障モード影響解析）の
情報資産から関連レコードを検索し、根拠を示しながら回答するエージェント型アプリケーションのモックアップを作る。

本仕様書は、DataRobotの `datarobot-agent-application`（Agentic Starter Application Template）をベースに、
エージェントフレームワークとして **NAT（NVIDIA NeMo Agent Toolkit）** を採用して実装することを前提に書かれている。

成功の判定基準は、技術者が工程名や不良現象に関する曖昧な日本語の質問を入力した際に、関連性の高いPFMEAレコード
（故障モード・影響・原因・現在の管理・推奨対策・リスク優先数RPN）を提示できることである。本フェーズではモックアップとして
動くことを優先し、認証・権限制御・本番運用上の堅牢性は対象外とする。

## 2. スコープ

### スコープ内
- NATワークフロー（`workflow.yaml`）として、ユーザーの質問を受け取り、PFMEAデータを検索して回答を生成するエージェントを1つ実装する。
- PFMEAデータに対する検索ツールをNATの関数（ツール）として実装する（後述の検索仕様に従う）。
- `datarobot-agent-application` のローカル開発フロー（`dr start` ウィザード、`task` コマンド）でローカル実行・動作確認ができる状態にする。
- 同梱のフロントエンド（React/Vite）でチャットUIから質問・応答できる状態にする。
- 提供する `pfmea_data.json` をそのままデータソースとして使う（DB移行は対象外）。

### スコープ外（本フェーズでは対応しない）
- 本番デプロイ（Pulumiによるクラウドプロビジョニング）・DataRobotプラットフォームへの実デプロイ
- OAuth等の認証・権限制御
- Google Drive / Box など外部ドキュメントソースとの連携
- ベクトル検索・Embeddingによる意味的検索（将来の拡張候補としてのみ言及。後述）
- PFMEAデータの追加・編集機能（読み取り専用の検索のみ）
- 複数言語対応（日本語のみを想定）

## 3. 前提とする土台（リポジトリ）

- ベーステンプレート: `https://github.com/datarobot-community/datarobot-agent-application`
- エージェントフレームワーク: **NAT (NVIDIA NeMo Agent Toolkit)**
  - NATテンプレートはエージェント定義・ツール・LLM設定を `workflow.yaml` で宣言的に記述する点が他フレームワーク
    （CrewAI / LangGraph / LlamaIndex）と異なる。Pythonの `llm` プロパティではなく `workflow.yaml` 内で
    LLMを定義する。
  - ツールは `functions` セクションに `_type: chat_completion` などとして定義し、ワークフローの `tool_list` で
    参照する。
  - LLMは `llms` セクションで定義し、DataRobot LLM Gatewayを使う場合は `_type: datarobot-llm-gateway` を指定する。
- 動作環境: macOS / Linuxのみ対応（Windowsの場合はDataRobot codespace、WSL、または仮想マシンを使用する）。
- ローカル起動は `dr start`（初回はテンプレート選択・clone・`.env`生成のウィザードが動く）、コンポーネント起動は
  `task` コマンド群を使う。

実装を始める前に、Claude Codeは上記リポジトリをclone（または `dr start` のウィザードでテンプレートを取得）し、
`README.md` および `docs/` 配下（特に `docs/mcp-server.md` など）の最新情報を必ず確認し、本仕様書との差分が
あれば本仕様書よりリポジトリの実際の構成を優先すること。テンプレートは更新頻度が高いため、ファイル名やディレクトリ構成が
本仕様書の記述と変わっている可能性がある。

## 4. データソース

別添 `pfmea_data.json` を使用する。固形製剤（錠剤）製造ラインの架空データで、構成は以下の通り。

```jsonc
{
  "meta": { "title": "...", "line": "...", "version": "...", "last_updated": "...", "note": "..." },
  "processes": [
    {
      "process_id": "P01",       // 工程ID（検索結果の紐付けキー）
      "process_name": "秤量",     // 工程名（日本語）
      "process_no": 10,           // 工程番号（10刻み）
      "description": "...",       // 工程の説明
      "equipment": "..."          // 使用設備
    }
    // ... P01〜P10（秤量・混合・造粒・乾燥・整粒・打錠・コーティング・外観検査・充填・包装）
  ],
  "fmea_records": [
    {
      "record_id": "F0101",                 // レコードID
      "process_id": "P01",                  // 紐付く工程ID
      "function": "...",                     // その工程に求められる機能
      "failure_mode": "...",                 // 故障モード（不良現象）
      "failure_effect": "...",               // 故障の影響
      "severity": 8,                         // 厳しさ（1-10）
      "potential_cause": ["...", "..."],     // 潜在的原因（配列）
      "occurrence": 3,                       // 発生度（1-10）
      "current_control": "...",              // 現在の管理（検出方法）
      "detection": 4,                        // 検出度（1-10、低いほど検出しやすい）
      "rpn": 96,                             // リスク優先数（severity × occurrence × detection ではなく、
                                              //   本データでは独自に算出済みの値をそのまま使用する）
      "recommended_action": "...",           // 推奨対策
      "keywords": ["...", "..."]             // 検索用キーワード（同義語・略称含む）
    }
    // ... 全50件、P01〜P10の各工程に4〜7件ずつ
  ]
}
```

注記：本データは検討用に作成した架空のサンプルであり、実在の製造ラインの情報ではない。

## 5. 検索仕様（ツールのロジック）

ユーザーは曖昧な日本語で質問する（例:「打錠工程で注意するべき不良現象は？」「異物混入のリスクが高い工程はどこ？」
「コーティングのムラを防ぐには？」「RPNが高い不良モードを教えて」）。検索ツールは以下の方針で実装する。

### 5.1 工程名の検出
質問文に工程名（秤量・混合・造粒・乾燥・整粒・打錠・コーティング・外観検査・充填・包装）または、その揺らぎ表現
（例:「計量」→秤量、「攪拌」→混合、「フィルムコート」→コーティング、「ブリスター」→充填）が含まれる場合、
対応する `process_id` を検出し、検索結果の絞り込み・優先付けに利用する。
シノニム辞書は実装側で用意してよい（過去の試作HTMLモックアップに実装例があるので参考にすること。本仕様書末尾の
「参考実装」を参照）。

### 5.2 キーワード抽出
日本語の分かち書き（形態素解析）は本フェーズでは必須としない。形態素解析器（例: Janome、SudachiPyなど軽量なもの）
が利用できる場合は導入して精度を上げてもよいが、依存追加が難しい場合は次の簡易方式でよい。
- 助詞・助動詞・一般的なストップワード（「の」「が」「を」「教えて」「ください」「について」等）を除去する。
- 残った文字列から、レコードの `keywords` ・`failure_mode` ・`potential_cause` ・`current_control` ・
  `recommended_action` に対して部分一致でスコアリングする。

### 5.3 スコアリングとランキング
- 工程IDが一致するレコードにはボーナススコアを与える。
- `keywords` フィールドとの一致は `failure_mode` 等の文章中の部分一致よりも重みを高くする。
- 「RPNが高い」「リスク優先」「重大」等の意図を検出した場合は、（工程が指定されていればその工程内で、
  指定がなければ全体で）RPN降順に上位を返す通常検索とは別の応答モードに切り替える。
- 該当レコードが0件の場合は、質問の言い換え例を提示する等、ユーザーが次の入力をしやすい応答にする。
- 上位5件程度を返却する（多すぎる場合はチャットUI上で冗長になるため）。

### 5.4 LLMによる回答生成
検索ツールが返した上位レコード（構造化データ）を、NATワークフローのLLM（`chat_completion` function）に渡し、
質問に対する自然文の要約回答を生成させる。その際、回答の根拠となったレコードのIDも併記させ、ハルシネーション
（検索結果に存在しない情報の創作）を避けるよう、system_promptで「検索結果に記載のない情報は答えない」ことを
明示的に指示すること。

## 6. NATワークフロー構成（実装イメージ）

`workflow.yaml` に以下の要素を定義する想定。実際のキー名・スキーマはNATおよびDataRobotテンプレートの
最新仕様に従うこと（本仕様書はイメージを示すものであり、最終的な記法はリポジトリのサンプル
`workflow.yaml` を確認して合わせること）。

```yaml
llms:
  datarobot_llm:
    _type: datarobot-llm-gateway
    model_name: azure/gpt-4o-mini   # 利用可能なモデルに合わせて変更
    temperature: 0.0

functions:
  pfmea_search:
    _type: <NATのカスタムツール関数を定義する型>
    description: >
      製剤工程（秤量・混合・造粒・乾燥・整粒・打錠・コーティング・外観検査・充填・包装）の
      PFMEAレコードを検索する。工程名や不良現象に関するキーワードを受け取り、
      関連度の高い順に故障モード・原因・対策等を返す。
    # 実装は Python 関数（例: agent_retrieval/custom_model/tool.py 等、
    # テンプレートの実際の配置に合わせる）を参照させる

  pfmea_assistant:
    _type: chat_completion
    llm_name: datarobot_llm
    system_prompt: |
      あなたは製薬工場の製剤工程に関する品質管理アシスタントです。
      ユーザーからの質問に対して、必ず pfmea_search ツールで検索した結果のみに基づいて回答してください。
      検索結果に含まれない情報を推測で答えてはいけません。
      回答には、根拠としたレコードID（例: F0601）を明記してください。
      該当する情報が見つからない場合は、その旨を伝え、質問の言い換え例を提案してください。

workflow:
  _type: <ワークフローのエントリポイントの型>
  tool_list:
    - pfmea_search
  # pfmea_assistant をトップレベルのエージェントとして設定
```

## 8. 動作確認シナリオ

実装後、以下の質問でローカル動作確認を行う（`task agent:cli -- execute --user_prompt '...'` 等、
テンプレートのCLI実行コマンドを使用）。

1. 「打錠工程で注意するべき不良現象は？」
   → P06（打錠）に紐づくレコード（F0601〜F0604）が関連度上位に出ること。
2. 「異物混入のリスクが高い工程はどこ？」
   → 複数工程（P01, P02, P06等）の異物関連レコードが横断的に検索され、severityやRPNの高いものが優先されること。
3. 「コーティングのムラを防ぐには？」
   → F0701（コーティング不均一）が検出され、`recommended_action` の内容が回答に反映されること。
4. 「RPNが高い不良モードを教えて」
   → RPN降順で全体上位（F0201: RPN160等）が提示されること。
5. 該当しない質問（例:「今日の天気は？」）
   → 検索結果0件として、関連情報が見つからない旨と質問の言い換え例が返ること。

その後、フロントエンド（React/Vite）のチャットUIからも同様の質問を入力し、レスポンスが画面に表示されることを
確認する。
